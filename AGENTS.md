## Architecture Rules

Non-negotiable. If a task seems to require violating one, stop and ask.
(Code layout and component map: `.planning/codebase/STRUCTURE.md`, `ARCHITECTURE.md`.)

- **Lemmas** deduplicate cards across word forms. Cards reference `lemma_id`. Never create a second card for the same lemma.
- **One card type** with a `card_kind` discriminator (`'word' | 'sentence'`).
- **Drizzle ORM is the single source of truth for schema.** DB migrations and TS types both derive from it.
- **Provider boundaries:** TTS, image search, LLM, and audio storage are accessed only through the provider files; implementations are swappable. A new external service requires an ADR before adding a provider.
- **AI usage:** AI generates only content unique to the user's state (i+1 sentences, card fields). Curated content is never AI-generated at runtime.
- **Every LLM output is validated through a Zod schema** before use. No untyped LLM JSON in app code.
- **No silent fallbacks.** Malformed LLM JSON throws or logs loudly — never quietly defaults.
- **Specific error types**, not generic `Error`. Each provider defines its own error union.
- **Prompts live in `prompts/`** as versioned files, not inline strings.

## Code Conventions

Style (formatting, file naming, import order) is enforced by ESLint + Prettier + TS
strict mode — just run `pnpm lint` / `pnpm format`. Full rules: `.planning/codebase/CONVENTIONS.md`.
The judgment calls tooling can't enforce:

- **Discriminated unions over flag fields.** Constants over magic numbers/strings.
- **Comment only when the WHY is non-obvious** (incl. why a "future use" schema field exists now); default to none.
- **No premature abstraction.** A function called twice doesn't need extracting.
- No `any`, `as` casts, or `// @ts-ignore` without an explanatory comment.

## Communication Style

- **Terse by default.** No preamble, no recap, no filler.
- Bullets and tables over prose.
- End every response with one line: `Status: ✅ Done | 🛑 Awaiting input | ⚠️ Partial`
- Inline markers: 🛑 blocking question, ❓ decision needed, ⚠️ warning.
- Verbose only for: plans awaiting approval, ADRs, debugging sessions.

## Testing Policy

Test what would hurt to break; skip the rest. Vitest; patterns and layout in `.planning/codebase/TESTING.md`.

- **Always:** pure logic in `lib/` and domain modules; data boundaries (Zod AI-output validation, DB serialization, audio cache dedupe); critical flows sparingly ("generate card → approve → appears in review queue").
- **Don't:** UI render details, external API responses (mock the provider boundary), AI-generated content quality.

## Working with GitHub

- **Issues:** Use `gh issue create` for any bug/feature/ADR-needed found outside the current task. **Don't silently fix unrelated bugs** — file it, link it, decide later. Labels: `bug`, `feature`, `chore`, `adr`, phase (`v0`/`v1`/`v2`), size (`size/s`…`size/xxl`).
- **Commits:** Conventional Commits via commitlint (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`). Explain _why_, not just _what_. Branch off `main`, PR back to `main`.
- **PRs via `gh`:** `gh` isn't logged in (the sandbox proxy only auths `git`). Before any `gh` call, load the token from the gitignored `.env`: `export GH_TOKEN=$(grep '^GH_TOKEN=' .env | cut -d= -f2-)`. Never commit the token value.
- **Releases:** `release-please` runs on every push to `main` — merge its PR to ship. Don't edit `CHANGELOG.md` manually.

## What to Ask About Before Doing

File an issue with the `adr` label and pause for:

- Adding a new external service or provider
- Schema changes that affect existing data
- Reorganizing folder structure or feature boundaries
- Adding a new dependency (especially large or copyleft)
- Changing a non-goal (see `docs/specs/v0.md`)
- Anything that violates an existing ADR
- Building anything from `docs/ideas.md`

## Glossary Quick-Reference

(Full glossary at `docs/glossary.md`.)

- **Lemma** — dictionary headword; mastery anchor. **Form** — surface form of a lemma (`falo` is a form of `falar`).
- **FSRS** — Free Spaced Repetition Scheduler. **i+1** — input one step beyond current ability.
- **CEFR** — Common European Framework (A1–C2). **L1** — user's native language(s).
