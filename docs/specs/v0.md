# V0 Spec — Brazilian Portuguese Learning App

**Status:** V0 — Personal Daily Driver
**Audience:** Solo founder + AI coding tools

---

## Vision

An AI-native Fluent Forever-style language learning app. Brazilian
Portuguese first. Personal daily driver, possible public product later.

The wedge: card creation drops from ~30 seconds to ~5 seconds because
AI proposes everything (image candidates, sentence, audio, gendered
form, register, usage context) and the user picks/approves. The user
keeps personalization (picks the image, edits the sentence) without
the grind that kills most FF-method learners.

V0 must be the founder's daily driver before anything else happens.

---

## Principles

1. **AI proposes, user disposes.** No card is silently committed.
   Every card passes through a picker + edit screen.
2. **AI is a specific tool, not a default.** AI for content unique to
   the user (i+1 sentences). Curated content (when added) is authored,
   never AI-generated at runtime.
3. **i+1 always.** Sentences use words the user knows plus exactly one
   target. Pre-made decks failed this; this is the differentiator.
4. **Audio is non-negotiable.** Every card has TTS audio. Pronunciation
   is learned from listening, not from IPA.
5. **No translation crutch.** Cards use images, not English glosses.
6. **Personal-first.** Founder's daily driver before strangers see it.
7. **Built to extend, not yet extensive.** Schema flexible, no plugins,
   no premature features.
8. **Permissive licensing only.** No GPL, AGPL, CC-BY-SA. Proprietary
   accepted where quality warrants AND dependency is swappable.

## V0 Scope

### Card types

**Word card** — user types a single word, pipeline produces a card.
Fields:

- `headword` (display form, may equal lemma or be a form like _falando_)
- `lemma_id` (FK to lemmas — canonical dictionary form)
- `gendered_form` (e.g., "a feijoada", "o livro" — AI-generated)
- `gender` (masculine / feminine / common — visual treatment TBD at UI time)
- `stress_marker` (string indicating stressed syllable, e.g., "fei.JO.a.da")
- `usage_context` (1–3 short phrases, e.g., "restaurante, almoço")
- `register_tag` (formal / neutral / informal / slang / vulgar)
- `sounds_like` (optional cognate/memory hook, may be null)
- `image_url`, `image_attribution`
- `audio_clip_hash` (FK to audio_clips by content hash)
- `sentence_pt` (chosen i+1 sentence)
- `sentence_audio_clip_hash`
- `sentence_gloss_en` (hidden by default; available for future hover-translation)

**Sentence card** — user pastes a Portuguese sentence, pipeline
produces a card with image + audio. Fields:

- `sentence_pt` (the user's sentence)
- `image_url`, `image_attribution`
- `sentence_audio_clip_hash`

Both card types share: `card_kind` enum, FSRS state, deck (one default
deck at V0), user, timestamps.

### Pipeline

**Word path:**

1. User types a word
2. AI extracts: lemma, gender, gendered_form, stress_marker,
   usage_context, register_tag, sounds_like, image search query,
   4 i+1 sentence candidates
3. Image search returns 4 candidates from search query
4. User picks image, picks sentence, edits anything
5. TTS generates audio for the lemma and chosen sentence on save
6. Card enters FSRS queue with `due_at = now`

**Sentence path:**

1. User pastes a sentence
2. AI extracts: image search query, optional gloss
3. Image search returns 4 candidates
4. User picks image, edits anything
5. TTS generates audio for the sentence on save
6. Card enters FSRS queue with `due_at = now`

### Review loop

- FSRS scheduler via `ts-fsrs`
- Daily review queue, audio-first presentation
- Standard ratings: again / hard / good / easy
- Review log records rating, duration, state transitions
- One card style at V0 (front: image + audio; back: full content)

### What's in V0 explicitly

- Card generation (word + sentence inputs)
- 4-option image picker, 4-option sentence picker
- Edit-before-save screen
- "Learn this" word capture from anywhere in the app
- FSRS daily review with audio
- Cloud sync (cloud is source of truth, clients are caches)
- TTS audio caching (content-hash dedup)
- Single user, hardcoded — no auth
- Single dialect (BR Portuguese, neutral broadcast)
- Single formality (você + 3rd-person singular)
- Single TTS provider/voice (Narakeet, one chosen voice)
- Mobile (Expo Go for personal use) and web (RN Web in browser)

### What's deferred

Everything not in the list above. Anything you want to add gets filed
in `docs/ideas.md` first, then ADR, then implementation.

## Schema (V0)

Six tables. Postgres + Drizzle ORM.

```sql
users (
  id              UUID PRIMARY KEY,
  email           TEXT,                -- nullable at V0 (no auth)
  display_name    TEXT,
  audio_speed     REAL DEFAULT 1.0,
  created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMP NOT NULL DEFAULT NOW()
)

lemmas (
  id              SERIAL PRIMARY KEY,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  headword        TEXT NOT NULL,       -- canonical dictionary form
  created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
  UNIQUE (user_id, headword)
)

cards (
  id                       SERIAL PRIMARY KEY,
  user_id                  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  card_kind                TEXT NOT NULL CHECK (card_kind IN ('word', 'sentence')),
  lemma_id                 INTEGER REFERENCES lemmas(id),  -- NULL for sentence cards

  -- Word card fields (NULL for sentence cards)
  headword                 TEXT,
  gendered_form            TEXT,
  gender                   TEXT CHECK (gender IN ('masculine', 'feminine', 'common')),
  stress_marker            TEXT,
  usage_context            TEXT,
  register_tag             TEXT CHECK (register_tag IN
                            ('formal', 'neutral', 'informal', 'slang', 'vulgar')),
  sounds_like              TEXT,

  -- Shared fields
  image_url                TEXT,
  image_attribution        TEXT,
  audio_clip_hash          TEXT,             -- nullable; lemma audio
  sentence_pt              TEXT NOT NULL,
  sentence_audio_clip_hash TEXT NOT NULL,
  sentence_gloss_en        TEXT,             -- hidden by default

  -- FSRS state (merged into cards table at V0)
  due_at                   TIMESTAMP NOT NULL,
  stability                REAL NOT NULL DEFAULT 0,
  difficulty               REAL NOT NULL DEFAULT 0,
  last_reviewed_at         TIMESTAMP,
  reps                     INTEGER NOT NULL DEFAULT 0,
  lapses                   INTEGER NOT NULL DEFAULT 0,
  state                    TEXT NOT NULL DEFAULT 'new'
                           CHECK (state IN ('new','learning','review','relearning')),

  created_at               TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at               TIMESTAMP NOT NULL DEFAULT NOW()
)

reviews (
  id              SERIAL PRIMARY KEY,
  card_id         INTEGER NOT NULL REFERENCES cards(id) ON DELETE CASCADE,
  reviewed_at     TIMESTAMP NOT NULL DEFAULT NOW(),
  rating          TEXT NOT NULL CHECK (rating IN ('again','hard','good','easy')),
  duration_ms     INTEGER,
  state_before    TEXT NOT NULL,
  state_after     TEXT NOT NULL,
  scheduled_days  REAL NOT NULL
)

audio_clips (
  content_hash    TEXT PRIMARY KEY,     -- SHA-256(text + provider + voice_id)
  text            TEXT NOT NULL,
  provider        TEXT NOT NULL,        -- 'narakeet' at V0; expandable
  voice_id        TEXT NOT NULL,
  storage_url     TEXT NOT NULL,
  duration_ms     INTEGER,
  generated_at    TIMESTAMP NOT NULL DEFAULT NOW()
)

pending_cards (
  id              SERIAL PRIMARY KEY,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  input_text      TEXT NOT NULL,        -- the word or sentence the user entered
  input_kind      TEXT NOT NULL CHECK (input_kind IN ('word', 'sentence')),
  status          TEXT NOT NULL CHECK (status IN
                   ('pending', 'generating', 'ready_for_review', 'discarded')),
  draft_json      JSONB,                -- full draft including 4 candidates
  created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMP NOT NULL DEFAULT NOW()
)
```

**Schema notes:**

- Lemmas exist at V0 to prevent duplicate cards from different forms
  of the same word. _falando_ → lemma _falar_; second card creation
  for _falar_ reuses the lemma.
- FSRS state lives on `cards` table at V0 for simplicity. Can split
  out at V2 if access patterns demand.
- `audio_clips.content_hash` includes provider + voice_id so multiple
  providers can coexist without collision (architecture supports
  future provider-switching feature).
- `pending_cards` survives connection drops mid-flow.

## Stack

- **App:** Expo + React Native + React Native Web (single codebase)
- **Language:** TypeScript strict + `noUncheckedIndexedAccess`
- **DB:** PostgreSQL + Drizzle ORM (single source of truth)
- **SRS:** `ts-fsrs` (MIT)
- **TTS:** Narakeet (V0 primary, async job model). Provider boundary
  exists; future providers (ElevenLabs, Azure, Coqui) plug in.
- **LLM:** Claude or GPT — pinned in implementation, ADR required
- **Image search:** Google Custom Search or Bing — pinned in implementation
- **Audio storage:** S3-compatible — pinned in implementation
- **Validation:** Zod (every AI output, every API boundary)
- **Testing:** Vitest (unit), Detox or Maestro TBD (flow smoke tests)

---

## Architecture Rules

1. **Cloud is source of truth.** Clients are caches. Optimistic local
   writes, synced on reconnect. Last-write-wins at V0.
2. **Provider boundaries** for TTS, LLM, image search, audio storage.
   Application code imports interfaces, not SDKs.
3. **`Result<T, E>` at every data boundary.** No silent fallbacks.
4. **Prompts are versioned files** in `prompts/`. Not inline strings.
5. **AI output validated through Zod** before use. Never trust raw JSON.
6. **No copyleft licenses.** GPL, AGPL, CC-BY-SA excluded.

## Estimation Convention

T-shirt sizing on all issues and milestones:

- **S** — a few hours, single sitting, low complexity
- **M** — a day or two, single concern, requires thought
- **L** — several days, multiple concerns. Break down before scheduling.
- **XL** — substantial, multiple unknowns. Break down before scheduling.
- **XXL+** — grand ideas, brainstorm tier. Captured in `docs/ideas.md`,
  not estimated, not scheduled. Reevaluated when tech/tools change.

L and XL items get decomposed into S/M pieces before work starts.
XXL+ items stay in `docs/ideas.md` until something changes their
tractability — then they get decomposed and moved to active work.

---

## Success Criteria

V0 is done when all five are true:

1. Add a word, get a usable card, approve it in <30 seconds.
2. AI sentences are i+1 (use only known vocabulary plus the target).
3. Complete 7 consecutive days of reviews without using prior tool.
4. TTS audio quality sufficient to learn pronunciation from.
5. Card-gen pipeline produces usable card on 80%+ of inputs without
   hitting an edge case.

V0 ships when all five are true. Not before.

---

## First Milestone (Recommended)

Before building any UI, validate the core pipeline:

### CLI spike (size: S–M)

1. Type a Portuguese word into a CLI script
2. Get back JSON with: image search results, audio file (Narakeet),
   lemma identification, 4 sentence candidates, all card fields
3. Listen to audio, look at images
4. Decide if quality is good enough

If quality is good, V0 has a clear path. If not, the pipeline needs
work before any UI investment.

## Non-Goals

- Native SwiftUI iOS, ever (Expo + RN Web only)
- Native Android codebase, ever (Expo Android build is the path)
- Tutor chatbot (AI is in card pipeline only)
- Communication platform (no messaging, comments, posts, follows-as-feed)
- Curriculum-driven instruction (frequency list provides structure;
  user controls pace)
- Translation-based learning (English glosses are an escape hatch)
- App Store at V0 or V1 (V2+ via EAS Build)
- Built on proprietary structured data (no FF curated frequency list)
- Vendor-locked TTS (architecture keeps swappable)
