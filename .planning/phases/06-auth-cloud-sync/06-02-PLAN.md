<objective>
Establish the database layer for Supabase Auth: a Postgres trigger that auto-creates a `public.users`
profile row whenever Supabase Auth inserts into `auth.users`, and an explicit, user-confirmed decision
on the RLS roadmap deliverable (which RESEARCH Open Question 4 flags as in tension with ADR 0007 rule 3).

Purpose: The trigger keeps existing FK relationships intact when real users sign up (RESEARCH Pattern 8).
The RLS decision is a roadmap-listed Phase 6 deliverable ("Row-level security on all DB tables") that
RESEARCH shows is a silent no-op under the postgres/BYPASSRLS role — it must be resolved with the user,
not dropped or stubbed.
Output: A raw SQL migration with the trigger, and a recorded RLS decision.
</objective>

It defines `public.handle_new_user()` as a `security definer` function with `set search_path = ''` that inserts into `public.users (id, email, display_name)` using `new.id`, `new.email`, and `coalesce(new.raw_user_meta_data->>'display_name', split_part(new.email, '@', 1))` for display_name.
